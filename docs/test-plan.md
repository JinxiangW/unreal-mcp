# 功能与 Robustness 测试方案

这份文档定义：

1. 功能完成后的基础功能验证
2. 功能完成后的 robustness 验证

原则：

- 功能测试先证明“能用”
- robustness 测试再证明“不会轻易坏”
- 任何高层命令都至少覆盖：成功路径、错误路径、重复执行、边界输入

## 测试分层

### 1. Smoke Test

目的：

- 确认链路活着

适用对象：

- 每次重启编辑器后
- 每次切换执行后端后

最小检查：

- MCP 端口可连
- 基础 tool 可调用
- 当前关卡可读

### 2. Functional Test

目的：

- 确认该功能满足预期业务结果

最小要求：

- 至少一条真实成功路径
- 必须回读关键状态

### 3. Robustness Test

目的：

- 确认该功能在重复、边界、异常输入下不容易失稳

必须包含：

- 批量规模测试
- 部分失败测试
- 结果回执正确性测试

### 4. Recovery Test

目的：

- 确认失败后系统还能继续工作

## 各域测试要求

### Scene

#### Functional

- 灯光创建后回读：
  - location
  - rotation
  - intensity
  - mobility
  - intensity unit

- 后处理修改后回读：
  - override 标志
  - 关键数值字段

#### Robustness

- 同一高层命令连续执行 3 次
- `replace_existing=True/False` 都测
- 非法 actor 名称
- 非法目标点 / 半径 / count
- 编辑器已有同名 Actor 的情况

#### Recovery

- 命令失败后再次执行简单读取命令，确认 MCP 仍然活着

### Asset

#### Functional

- 创建资产后确认 `.uasset` 落盘或能被 UE 读到
- 更新属性后回读确认
- 导入后确认返回的 `imported_object_paths`

#### Robustness

- 重复导入同一资源
- 目标路径已存在
- 无效源文件路径
- 不支持的 asset type
- 属性里混合基础类型和资产引用
- 10 / 50 / 100 级别批量操作
- 批量里混入部分非法条目

#### Recovery

- 导入失败后再次执行 `create_asset_with_properties`
- commandlet 失败后再次执行 commandlet，确认无残留状态阻塞

### Material

#### Functional

- 创建材质资产
- 创建材质实例
- 设置 scalar / vector / texture 参数并回读
- 参数名查询正确

#### Robustness

- 不存在的参数名
- 参数类型不匹配
- 不存在的纹理路径
- 重复设置同一个参数多次
- 10 / 50 / 100 级别批量参数更新
- 混合有效参数和无效参数名

#### Recovery

- 参数设置失败后，再做一次参数名读取和其他参数设置，确认链路仍可用

### Material Graph

#### Functional

- 读取图
- 创建节点
- 连线
- 重新读取图确认变化

#### Robustness

- 连不存在的节点
- 连不兼容插槽
- 重复创建同名参数节点
- 图为空 / 图已有复杂内容时都能处理
- 节点批量创建和批量连线测试
- 大图情况下的读回验证

## 批处理专项测试

所有支持批处理的功能，必须补这四组测试：

### 1. 小批量

- 10 个对象

### 2. 中批量

- 50 个对象

### 3. 大批量

- 100 个对象起步

### 4. 部分失败

- 让 5% 到 20% 的对象故意失败
- 观察是否还能返回正确 summary 和 item 级结果

## 结果校验机制

所有高层命令最终都应具备：

- `post_state`
- `verification.verified`
- `verification.checks`

批处理额外要求：

- `summary.requested`
- `summary.succeeded`
- `summary.failed`
- `summary.verified`
- `items[]`

详细设计见：

- `docs/verification.md`

#### Recovery

- 图编辑失败后仍能继续读图和做其他图操作

## 每个新高层命令的最小测试模板

### A. 成功路径

- 输入一组标准参数
- 确认返回 `success`
- 回读关键结果

### B. 重复执行

- 连续执行两到三次
- 确认没有名称冲突、残留对象、状态漂移

### C. 错误输入

- 缺必填参数
- 错路径
- 错名字
- 错类型

### D. 恢复性

- 紧接着执行一个简单读取命令
- 确认端口、MCP、当前域执行链仍正常

## 自动化建议

### 当前阶段建议保留的自动化层次

- Python 侧 smoke + functional 脚本
- commandlet 导入脚本
- 少量真实项目资源的真实环境回归

### 当前不建议过早做的

- 大而全的端到端全域自动化
- 多域共享一个超长回归脚本

原因：

- 目前架构还在拆分阶段
- 先做小而稳的域级回归更有效

## 当前推荐的测试顺序

1. 先做 smoke
2. 再做该域功能测试
3. 再做重复执行测试
4. 再做错误输入测试
5. 最后做恢复测试

## 通过标准

一个功能只有同时满足下面条件，才算完成：

- 至少一条真实成功路径通过
- 至少一条错误路径通过
- 至少一次重复执行没有异常
- 失败后系统仍可继续执行其他命令
- 关键状态已回读验证
